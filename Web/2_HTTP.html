HTTP

프로토콜이란

사람과 사람이 대화(통신)할때 서로 이해할 수 있는 언어, 공용된 언어를 사용해야 전 세계 사람과 대화할 수 있다.

컴퓨터와 컴퓨터도 통신(대화)할때 서로 이해할 수 있는 언어, 공용된 언어를 사용해야 통신할 수 있다.

프로토콜은 컴퓨터간 통신할때 공용 언어를 사용하자는 규칙, 규약

즉, 컴퓨터간 원활한 통신을 위해 약속된 규칙

내가 들어본건 HTTP, FTP 정도. 그 외에도 많지만 우선 HTTP에 집중하자

HTTP

HyperText Transfer Protocol

웹에서 데이터를 주고받을 수 있는 프로토콜

요청과 응답 형태

클라이언트 - 서버 구조

무상태성 (statekess) - 서버가 클라이언트의 상태를 저장하지 않는다.
요청 시 마다 필요한 정보를 다 보내야 한다.
아무 서버나 호출해도 된다. - 확장성 좋음, 특정 서버를 기다리지 않아도 된다.

비연결성 (connectionless) - 연결을 지속하지 않는다.
클라이언트가 요청 후 응답을 받으면 연결을 끊어버린다 - 서버의 자원을 필요할때만 사용한다.
다만, 요청 시 마다 계속 연결을 해야하기 때문에 추가적인 css, js 등을 받아야 하기에 비효율적이다.
'HTTP 지속 연결'로 해결 - 연결이 이뤄지고 모든 자원을 받을때까지 기다렸다가 연결을 끊는다.

서버와 클라이언트

클라이언트

서버의 서비스를 제공받아 사용하는 장치 또는 프로그램

서버에게 요청

데스트탑, 노트북, 스마트폰, 태블릿 등

프로그램으로는 웹 브라우저

서버

서비스를 제공하는 컴퓨터

클라이언트에게 응답

HTTP 메시지

클라이언트와 서버의 소통은 메시기로 이루어진다.

요청 메시지

공백 제외 3가지 부분 - Start line, Headers, Body

Start line = GET /test.html HTTP/1.1 => [HTTP Method] [Request Target] [HTTP Version]

Headers = 해당 요청에 대한 추가 정보를 Key:Value 형태로 담고있다.
Host: 요청하려는 서버의 호스트
User-agent: 클라이언트 프로그램 정보
Accept: 클라이언트가 받아들일 수 있는 MIME 타입
Authorization: 인증 토큰 정보
등등

Body = 서버로 전달되는 실제 데이터
가장 보편적으로 쓰이는 GET, POST, PUT, DELETE 에서
POST, PUT 은 Body에 데이터를 담아서 보낸다.
GET, DELETE 도 담을 수 있지만 권장되지 않는다.

GET - 조회
POST - 추가
PUT - 수정
DELETE - 삭제
PATCH - 부분 수정
OPTIONS - 허용된 통신 옵션 요청 후 허가된 요청에 한해서 전송하기 위한 보안상의 목적
발생 조건이 따로 있음 - SOP, CORS와 연관이 있네
추후 서버 공부할 때 다시 보자

응답 메시지

공백 제외 3가지 부분 - Status line, Headers, Body

Status line = HTTP/1.1 200 OK => [HTTP Version] [Status Code] [Status Text]

Headers
Status Code: 응답 상태 코드
Content-Type: 응답 바디의 타입
Content-Length: 응답 바디의 길이
Set-Cookie: 쿠키 정보
등등

Body = 응답 데이터

응답코드
1XX - 임시 응답 - 거의 만날일 없음(사실 잘 이해가 안감, 어떤 상황인지)
2XX - 성공
3XX - 리다이렉트 - 자동으로 앞에 m. 붙어서 모바일 페이지 보여주는게 이거네
4XX - 클라이언트 오류
5XX - 서버 오류
